Why You Should Stop Using useEffect for Data Fetching

React Query, SWR, and the new use() hook have changed the game. Here's why reaching for useEffect is almost never the right call anymore.

For years, the useEffect + useState combo was the standard recipe for fetching data in React. You'd write something like the snippet below, ship it, and call it a day. But that pattern comes with a surprising number of footguns — race conditions, missing loading states, no cache, duplicate requests — and the React ecosystem has quietly moved on. 

// ❌ The old pattern — more problems than it looks
useEffect(() => {
  fetch('/api/user')
    .then(r => r.json())
    .then(data => setUser(data));
}, []);

What's Wrong with useEffect for Data Fetching?

The short answer: nothing and everything. useEffect works. It will always work. But it was never designed to be a data-fetching primitive. It's a tool for synchronizing your component with an external system — not for managing async state, caching, deduplication, or background re-fetching.

Here's a non-exhaustive list of things you have to handle yourself:

  • Race conditions — two requests fire, the slower one resolves last and overwrites fresh data.
  • No caching — every mount triggers a fresh network request even if the data hasn't changed.
  • No deduplication — if three components need the same data, you get three requests.
  • Stale closures — useEffect closures can reference outdated state.
  • No retry logic — a flaky network means your users see a broken UI forever.

"The best code is the code you don't have to write." Managing async data correctly in useEffect adds hundreds of lines of boilerplate across a real app.

Enter React Query (TanStack Query)

React Query handles all of the above automatically. You declare what data you need, and the library handles how to get it, cache it, keep it fresh, and synchronise it across components. 

// ✅ The React Query way
import { useQuery } from '@tanstack/react-query';

function UserProfile() {
  const { data, isLoading, error } = useQuery({
    queryKey: ['user'],
    queryFn: () => fetch('/api/user').then(r => r.json()),
  });

  if (isLoading) return <Spinner />;
  if (error)     return <Error />;
  return <Profile user={data} />;
}

With four lines you get: automatic caching, background re-fetching on window focus, deduplication, request cancellation, retry logic, and stale-while-revalidate semantics. For free.

What About SWR?

SWR (from Vercel) follows the same philosophy but with a more minimal API. It's a great choice for smaller projects or when you're already in the Next.js ecosystem.

The New use() Hook

React 19 introduces a native use() hook that lets you await a Promise directly inside a component, integrating with React's Suspense boundary. It's not a full data-fetching solution on its own, but combined with server components and a good caching layer, it represents the direction React is heading.

When Is useEffect Still the Right Tool?

Plenty of legitimate cases remain: setting up event listeners, integrating with third-party libraries, synchronising with browser APIs (e.g., document.title), or starting an imperative animation. If you're not fetching data over a network, useEffect is probably still your friend.

The takeaway: the next time you instinctively reach for useEffect to fetch data, pause and ask whether React Query or SWR would serve you better. In nine out of ten cases, they will — and your components will be cleaner for it.
