Building a Design Token Pipeline
That Actually Scales

After three failed attempts, I found a design-token workflow that keeps designers and engineers in sync — without the constant frustration.

Design tokens — the named, platform-agnostic values that encode a design system's visual decisions — sound simple on paper. A colour is color.brand.primary. A spacing step is spacing.4. Easy. The hard part is the pipeline: how do you get those tokens from a designer's Figma file into production code, across multiple platforms, without things drifting apart the moment someone changes a hex value?

I've built (and broken) three different token pipelines over the last four years. This is what I learned.

Attempt #1 — The Spreadsheet

Our first approach was a shared Google Sheet. Designers would update values, engineers would copy them into CSS variables. This lasted about six weeks before it fell apart. The sheet was never up to date, nobody knew who owned it, and the "source of truth" was actually whichever engineer last ran a find-and-replace across the codebase.

Attempt #2 — Figma Variables → Manual Export

Figma's Variables feature (launched in 2023) felt like the answer. Designers define tokens in Figma, we export them as JSON via a plugin, check the JSON into the repo. Better! But still manual. A forgetful designer + a busy sprint meant the JSON and Figma would diverge again within weeks.

What Finally Worked: Style Dictionary + a CI Hook

The breakthrough was treating tokens like code. Here's the pipeline we use now:

  1. Designers work in Figma with Variables. A Figma webhook fires on every publish event.
  2. A lightweight GitHub Action receives the webhook, runs a Figma REST API export, and opens a PR with updated token JSON.
  3. Style Dictionary transforms the JSON into platform-specific outputs: CSS custom properties, iOS Swift enums, Android XML resources, and a TypeScript constants file.
  4. Changesets document what changed. Engineers review the PR, merge, and everything propagates automatically.
// tokens/color.json (simplified)
{
  "color": {
    "brand": {
      "primary":   { "value": "#f0a500", "$type": "color" },
      "secondary": { "value": "#0e0e13", "$type": "color" }
    },
    "text": {
      "default": { "value": "{color.neutral.900}", "$type": "color" },
      "muted":   { "value": "{color.neutral.500}", "$type": "color" }
    }
  }
}

The Key Insight: Semantic Layers

Raw values (#f0a500) should never appear in your components. Instead, you use semantic tokens that describe intent:

  • color.action.primary → maps to color.brand.primary
  • color.text.default → maps to color.neutral.900
  • spacing.component.button.paddingX → maps to spacing.5

When you rebrand, you only update the alias layer. Components don't need to change. This is the difference between a token system that scales and one that doesn't.

Tokens are not just about consistency — they're about the rate of change. A semantic layer lets you make big design decisions without touching component code.

Tooling Checklist

  • Style Dictionary — token transformation, highly recommended
  • Tokens Studio for Figma — richer than native Figma Variables, exports W3C-spec JSON
  • GitHub Actions — automate the export and PR creation
  • Changesets — semantic versioning for your token package

The magic isn't any single tool — it's removing humans from the sync loop. When tokens flow automatically from Figma to code, design drift stops being a cultural problem and starts being an engineering problem you can actually solve.
